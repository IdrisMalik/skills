# PayPal REST APIs

Use this reference when the task involves server-side PayPal REST calls, endpoint selection, REST client design, webhook handling, or production-readiness review. First inspect existing routes, REST wrappers, environment variable names, persisted PayPal IDs, webhook handlers, and tests before proposing changes.

## Base URLs

| Environment | Base URL |
|---|---|
| Sandbox | `https://api-m.sandbox.paypal.com` |
| Live | `https://api-m.paypal.com` |

Keep sandbox and live credentials, webhook IDs, URLs, and account IDs separate.

## OAuth access tokens

Server-side REST calls use OAuth 2.0 client credentials.

```http
POST /v1/oauth2/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
```

Rules:

- Cache access tokens until near expiry.
- Store client secrets only on the server.
- Never return OAuth access tokens to browser code.
- Log PayPal debug IDs from failed responses for support and reconciliation.
- Use `PayPal-Request-Id` for idempotent create/capture/refund/payout operations where supported.

## Browser-safe client token for SDK v6 features

Most JavaScript SDK v6 checkout flows authenticate with a browser-safe `clientId`. Some flows, such as Fastlane or supported vaulting flows, require a browser-safe client token generated by the server.

Use the OAuth token endpoint with `response_type=client_token` and domain binding where required by the selected PayPal SDK flow.

```http
POST /v1/oauth2/token
Authorization: Basic base64(client_id:client_secret)
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&response_type=client_token&domains[]=example.com
```

Return only the browser-safe token field expected by the frontend. Do not return the client secret or a normal server OAuth bearer token.

## Minimal TypeScript REST client

```typescript
const PAYPAL_BASE = process.env.PAYPAL_ENV === "live"
  ? "https://api-m.paypal.com"
  : "https://api-m.sandbox.paypal.com";

async function getAccessToken(): Promise<string> {
  const credentials = Buffer.from(
    `${process.env.PAYPAL_CLIENT_ID}:${process.env.PAYPAL_CLIENT_SECRET}`
  ).toString("base64");

  const response = await fetch(`${PAYPAL_BASE}/v1/oauth2/token`, {
    method: "POST",
    headers: {
      Authorization: `Basic ${credentials}`,
      "Content-Type": "application/x-www-form-urlencoded"
    },
    body: "grant_type=client_credentials"
  });

  const payload = await response.json().catch(() => ({}));
  if (!response.ok) {
    throw new Error(`PayPal OAuth failed: ${payload.error_description || payload.error || response.status}`);
  }

  return payload.access_token;
}

async function paypalRequest<T>(
  path: string,
  init: RequestInit & { requestId?: string } = {}
): Promise<T> {
  const token = await getAccessToken();
  const headers: Record<string, string> = {
    Authorization: `Bearer ${token}`,
    Accept: "application/json",
    ...(init.body ? { "Content-Type": "application/json" } : {}),
    ...(init.requestId ? { "PayPal-Request-Id": init.requestId } : {}),
    ...(init.headers as Record<string, string> | undefined)
  };

  const response = await fetch(`${PAYPAL_BASE}${path}`, { ...init, headers });
  const payload = await response.json().catch(() => ({}));

  if (!response.ok) {
    const debugId = response.headers.get("paypal-debug-id") || payload.debug_id;
    throw new Error(`PayPal API ${response.status}${debugId ? ` debug_id=${debugId}` : ""}: ${JSON.stringify(payload)}`);
  }

  return payload as T;
}
```

## Orders API v2: one-time checkout

### Create order

```typescript
async function createOrder(input: {
  userId: string;
  cartId: string;
  amount: string;
  currency: string;
  invoiceId: string;
}) {
  // Compute amount, currency, items, tax, discounts, shipping, and invoice ID from trusted server state.
  return paypalRequest<{ id: string; status: string }>("/v2/checkout/orders", {
    method: "POST",
    requestId: `order-${input.invoiceId}`,
    body: JSON.stringify({
      intent: "CAPTURE",
      purchase_units: [{
        invoice_id: input.invoiceId,
        custom_id: `${input.userId}:${input.cartId}`,
        amount: {
          currency_code: input.currency,
          value: input.amount
        }
      }],
      payment_source: {
        paypal: {
          experience_context: {
            shipping_preference: "NO_SHIPPING",
            user_action: "PAY_NOW",
            return_url: "https://example.com/paypal/return",
            cancel_url: "https://example.com/paypal/cancel"
          }
        }
      }
    })
  });
}
```

### Capture order

```typescript
async function captureOrder(orderId: string, expected: { amount: string; currency: string }) {
  const order = await paypalRequest<any>(
    `/v2/checkout/orders/${encodeURIComponent(orderId)}/capture`,
    { method: "POST", requestId: `capture-${orderId}` }
  );

  if (order.status !== "COMPLETED") {
    throw new Error(`Unexpected PayPal order status: ${order.status}`);
  }

  const capture = order.purchase_units?.[0]?.payments?.captures?.[0];
  if (!capture) {
    throw new Error("PayPal capture response did not contain a capture record");
  }

  if (
    capture.amount?.value !== expected.amount ||
    capture.amount?.currency_code !== expected.currency
  ) {
    throw new Error("PayPal capture amount/currency mismatch");
  }

  return {
    orderId: order.id,
    captureId: capture.id,
    status: capture.status,
    amount: capture.amount.value,
    currency: capture.amount.currency_code
  };
}
```

### Authorize and capture later

Use `intent: "AUTHORIZE"` when the business flow requires delayed capture.

Relevant endpoints:

- `POST /v2/checkout/orders/{order_id}/authorize`
- `GET /v2/payments/authorizations/{authorization_id}`
- `POST /v2/payments/authorizations/{authorization_id}/capture`
- `POST /v2/payments/authorizations/{authorization_id}/void`
- `POST /v2/payments/authorizations/{authorization_id}/reauthorize`

Verify authorization honor periods, reauthorization limits, account eligibility, and capture deadlines before implementing delayed capture.

## Payments API v2

Use Payments v2 for captures, authorizations, refunds, and payment-resource details.

Useful endpoints:

- `GET /v2/payments/captures/{capture_id}`
- `POST /v2/payments/captures/{capture_id}/refund`
- `GET /v2/payments/refunds/{refund_id}`
- `GET /v2/payments/authorizations/{authorization_id}`
- `POST /v2/payments/authorizations/{authorization_id}/capture`
- `POST /v2/payments/authorizations/{authorization_id}/void`

### Refund a capture

```typescript
async function refundCapture(input: {
  captureId: string;
  refundIdempotencyKey: string;
  amount?: string;
  currency?: string;
  note?: string;
}) {
  const body: Record<string, unknown> = {};

  if (input.amount && input.currency) {
    body.amount = {
      value: input.amount,
      currency_code: input.currency
    };
  }

  if (input.note) {
    body.note_to_payer = input.note;
  }

  return paypalRequest(`/v2/payments/captures/${encodeURIComponent(input.captureId)}/refund`, {
    method: "POST",
    requestId: input.refundIdempotencyKey,
    body: JSON.stringify(body)
  });
}
```

Refund by capture ID. Do not refund by PayPal order ID.

## Current REST API v1 product families

Do not mark an endpoint legacy only because it is under `/v1`. PayPal uses v1 paths for several active product families. Confirm current product availability and account eligibility before implementation.

### Subscriptions

Common endpoints:

- `POST /v1/catalogs/products`
- `POST /v1/billing/plans`
- `PATCH /v1/billing/plans/{plan_id}`
- `POST /v1/billing/subscriptions`
- `GET /v1/billing/subscriptions/{subscription_id}`
- `POST /v1/billing/subscriptions/{subscription_id}/cancel`
- `POST /v1/billing/subscriptions/{subscription_id}/suspend`
- `POST /v1/billing/subscriptions/{subscription_id}/activate`
- `POST /v1/billing/subscriptions/{subscription_id}/revise`

Rules:

- Create products/plans in a controlled server/admin flow.
- Store product ID, plan ID, subscription ID, user/customer ID, status, billing cycles, start time, and PayPal links.
- Verify subscription state server-side before granting access.
- Reconcile lifecycle events through webhooks.

### Payouts

Use Payouts to send money to recipients. Do not use it for normal buyer checkout settlement.

Core endpoints:

- `POST /v1/payments/payouts`
- `GET /v1/payments/payouts/{payout_batch_id}`
- `GET /v1/payments/payouts-item/{payout_item_id}`
- `POST /v1/payments/payouts-item/{payout_item_id}/cancel`

Rules:

- Use a stable `sender_batch_id` for idempotency.
- Store payout batch ID and payout item IDs.
- Reconcile item-level statuses, not just batch status.
- Verify funding source, recipient country support, fees, account approval, sanctions/compliance constraints, and operational limits.

### Webhooks

Use webhooks for asynchronous reconciliation.

Common endpoints:

- `POST /v1/notifications/webhooks`
- `GET /v1/notifications/webhooks`
- `POST /v1/notifications/verify-webhook-signature`
- `GET /v1/notifications/webhooks-events/{event_id}`
- `POST /v1/notifications/simulate-event`

Webhook rules:

1. Preserve raw request data if the framework requires it for signature verification.
2. Capture PayPal transmission headers and webhook ID for verification.
3. Verify the signature with PayPal or an official helper.
4. Deduplicate by event ID.
5. Re-fetch PayPal resource state for high-value or ambiguous transitions.
6. Apply idempotent database transitions.
7. Return 2xx only after durable persistence or queueing.

Common events to consider by scenario:

- Checkout/order: `CHECKOUT.ORDER.APPROVED`
- Capture: `PAYMENT.CAPTURE.COMPLETED`, `PAYMENT.CAPTURE.DENIED`, `PAYMENT.CAPTURE.REFUNDED`, `PAYMENT.CAPTURE.REVERSED`
- Subscription: `BILLING.SUBSCRIPTION.CREATED`, `BILLING.SUBSCRIPTION.ACTIVATED`, `BILLING.SUBSCRIPTION.CANCELLED`, `BILLING.SUBSCRIPTION.SUSPENDED`, `BILLING.SUBSCRIPTION.PAYMENT.FAILED`
- Disputes and payouts: use the event names enabled for the merchant account and product

Verify exact event names in the account's PayPal webhook catalog before final implementation.

### Disputes, transaction search, reports, invoicing, payment links

Use the relevant PayPal API family when the task is not checkout:

- Customer Disputes API v1 for dispute listing, detail retrieval, evidence, offers, and lifecycle actions.
- Transaction Search API v1 and reporting APIs for reconciliation/reporting.
- Invoicing API v2 for invoice workflows.
- Payment Links and Buttons APIs for managed links/buttons when a full custom checkout is unnecessary.

## Data model fields

For one-time checkout, store at minimum:

- Internal order ID
- Authenticated user/customer ID
- Cart/session ID
- PayPal order ID
- PayPal capture ID or authorization ID
- Invoice ID/custom ID/reference ID
- Amount and currency as strings/decimals
- PayPal status and internal status
- Environment: sandbox or live
- Idempotency keys
- Webhook event IDs processed
- Normalized PayPal response data or audit records
- Timestamps for created, approved, captured, fulfilled, refunded, disputed, cancelled

## Implementation tests

- Create order from server-side cart state.
- Approve with sandbox buyer.
- Capture on server.
- Verify amount, currency, ownership, and status.
- Receive and verify webhook.
- Retry browser approval callback.
- Retry webhook delivery.
- Attempt duplicate capture or duplicate fulfillment.
- Refund full amount.
- Refund partial amount if supported by business rules.
- Cancel checkout.
- Simulate failed/denied capture.
- Test subscription activation, cancellation, failed renewal, and access revocation if using subscriptions.
- Test payout batch/item status if using payouts.
